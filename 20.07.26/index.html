1 Приглашения сотрудников в ленте офферов диспетчера FIX UI

Бэкенд обновил уведомление и офферное сообщение о приглашении в компанию для CM/DM, но диспетчер не видел приглашение в /offers, хотя счётчик показывал «1 входящий». Полное расследование /decider выявило цепочку из трёх находок.

  • Краш /offers. GET /v1/dispatchers/offers/all — полиморфная лента: строки приглашений приходят с cargo: null и offer: null, а тип SentOfferListItem объявлял оба поля не-nullable. item.cargo.id падал с TypeError, ErrorBoundary ловил и показывал «Что-то пошло не так». Поля сделаны nullable, строки-приглашения отделены предикатом isOfferListItemWithCargo, приглашения — отдельным предикатом isCompanyInvitationOffer (по offer_kind === 'company_invitation').
  • Карточка + модалка принятия. Приглашения рендерятся отдельной зелёной карточкой (CompanyInvitationOfferCard) с полями компании/роли/статуса/дат и модалкой Accept/Decline.
  • Реальный блокер — не фронтенд. CM/DM — обычный диспетчер, отдельное лицо от владельца/директора компании, и у него физически нет company-аккаунта (localStorage['sarbon_company_app_token'] → null, подтверждено в реальной сессии). Путь принятия, который отвечает на staging (/v1/invitations/:id/accept), требует именно company-токен — то есть чтобы вступить в компанию, нужно уже быть её пользователем: замкнутый круг для приглашённого. Путь, который подходит по модели продукта (POST /v1/dispatchers/invitations/{accept,decline}), уже развёрнут в production и не требует company-сессии — но ему ни разу не передаётся token приглашения ни в ленте офферов, ни в пуше. Полная диагностика, включая пробы staging/production по HTTP-кодам (404 против 401) и разбор реальных payload'ов — в company-invitation-offers-backend-ticket.md в корне репозитория.
/offers → карточка «Приглашение в компанию»: NotifTest Co, роль «Директор», статус «Ожидает», даты получения/истечения.
Модалка приёма: те же поля + кнопки «Отмена / Отклонить / Принять»; закрытие модалки не считается отклонением.
Кнопки видимы и кликабельны, но пока без token в ответе бэкенда запрос гарантированно вернёт 401 Отсутствует X-User-Token — фронтенд обрывает клик до сети и показывает пояснение вместо бесполезного round-trip. Это не UX-заглушка «на будущее», это единственный честный вариант до правки бэкенда.
Редизайн строки уведомления: структурированные поля (компания/роль/от кого/получено) вместо одной строки текста, зелёный акцент вместо фиолетового, иконка Building2.

2 Email водителя в детальной карточке FEATURE

Тикет SARB-DISP: GET /v1/dispatchers/drivers/{id} отдаёт email, но детальная страница водителя его не показывала — поле было объявлено в типе, но не выведено ни одним компонентом. Добавлена строка «Email» рядом с телефоном, тем же паттерном (readField/renderValue), с тем же фолбэком «Нет данных» для пустого значения — без валидации формата, значение выводится как есть.

/drivers/:id — новая строка «Email» между телефоном и именем; для этого водителя показан фолбэк «Нет данных» (email не указан).

3 Админ · Рейсы и офферы: смена контракта бэкенда (snake_case) + переименование действия FIX

Бэкенд сменил формат ответа /admin/trips и /admin/offers с PascalCase на snake_case без предупреждения — все поля (Status, CreatedAt, AgreedPrice и т.д.) разом стали «нет данных», хотя данные реально приходили. Добавлен единый нормализатор normalizeAdminRowKeys/pascalToSnake на границе запроса (adminRowFields.ts), применённый и к чтению, и к местам, где старые PascalCase-ключи молча писались обратно в payload. Заодно — кнопка удаления рейса на самом деле является мягкой отменой (soft-cancel на бэкенде), поэтому она переименована в «Отменить рейс» (иконка Trash2 → Ban), включая все 15 локалей ключей messages.delete* → .cancel*.

/admin/trips: статус, согласованная цена, обе оценки, даты — все поля читаются корректно; действие теперь «Отменить рейс».
Панель «Детали рейса»: полная хронология (создан/начат/в пути/доставлен) без единого «нет данных» там, где данные реально есть.
/admin/offers: цена, тип, кто предложил, статус, дата — та же нормализация применена к списку офферов.

4 Название города в адресах: до запятой + аббревиатура FIX

Во всех местах, показывающих адрес отправления/назначения (my-cargos, all-cargos, offers, админка), теперь выводится название до первой запятой вместо либо голого города, либо полного адреса. По итогам двух уточнений: административное слово (область, viloyati, region) не отбрасывается, а аббревиатурируется (область → обл., viloyati → vil.), а ведущий маркер города (г. Ташкент) — отбрасывается. Логика — cityNameFromAddress/abbreviateGeoWords в addressDisplay.ts, подключена через summarizeCargoRoute ко всем спискам разом.

/all-cargos: «Актобе», «Ташкент», «Стамбул» — короткие, консистентные названия вместо полных адресов маршрута.

5 Excel-экспорт: обрезание колонок — и повторно расследованное китайское меню FIX

Экспорт из /all-cargos (кнопка «Excel») открывает вкладку XLSX прямо в приложении через встроенный в вендорный пакет excel-viewer движок x-data-spreadsheet. Жалоба была на два симптома; оба были «исправлены» одним коммитом сегодня утром — но при съёмке скриншотов для этого отчёта выяснилось, что только первый симптом был реально устранён.

  • Обрезание на колонке Z — исправлено и подтверждено. Пакет вычисляет row.len из данных, но никогда не передаёт col.len, поэтому сетка молча падала на дефолт в 26 колонок. Теперь col.len вычисляется явно (buildExcelViewData/resolveGridColumnCount) — видно на скриншотах ниже: колонки идут далеко за K, показаны реальные данные вплоть до «Обновлён»/«Нужно машин».
  • Китайское контекстное меню — утренний фикс не работал, найден настоящий корень. Утренний коммит звал x_spreadsheet.locale(lang) — выглядит как правильный API и даже задокументирован пакетом как таковой. При живой проверке через Playwright меню всё равно оказалось на китайском. Трассировка минифицированного бандла показала: реальный текстовый lookup идёт через внутренний хелпер $(path, dict), который полностью игнорирует состояние, которое устанавливает .locale(), а вместо этого на каждый вызов заново читает document.documentElement.getAttribute('data-excel-viewer-lang'), по умолчанию падая на zh_cn, если атрибут не выставлен. Подтверждено вручную: даже ручной вызов .locale('en') прямо перед правым кликом ничего не менял — меню оставалось китайским. Настоящий фикс — выставлять этот DOM-атрибут (renderSheet() в ExcelViewer.tsx), что и было сделано сейчас, с записью в bug.md о ложном первом фиксе.
Экспорт грузов: колонки A…K видны без обрезания, реальные данные (контакт, телефон, ID компании).
Та же таблица, прокрученная вправо: колонки E…O — «Создан», «Обновлён», «Нужно машин», «Температура» — ничего не обрезано.
Правый клик по ячейке: «Copy / Cut / Paste / Paste values only / Insert row / Delete row / …» — и панель формата теперь «Normal», а не «普通». Проверено на реальном клике, не на ручном оверрайде.

6 Архитектура раздела компании: устранение дублирования и мёртвого кода REFACTOR

По результатам /arch-аудита файловой и кодовой архитектуры кабинета компании (tooling-backed — madge/knip) выполнен EVOLVE-проход по трём находкам без изменения поведения:

  • Магнит связности. Surface-нейтральные хелперы (toRecord, toRecordArray, getEnvelopeData, getPageItems, prettifyKey, …) вынесены из adminHelpers.ts в новый src/utils/recordParsing.ts; admin-модуль ре-экспортирует их для 42 существующих вызовов — ноль изменений на стороне потребителей.
  • Дублирование парсинга сотрудников. parseCompanyEmployeesResponse/buildManagerOptions/ ManagerOption переехали из companyTrips.ts в companyEmployees.ts, где им и место (там же остальная логика сотрудников); shortId обобщён и перенесён в formatters.ts.
  • Мёртвый экспорт. Неиспользуемый COMPANY_TYPES из companyCreate.ts удалён (нулевые потребители подтверждены, включая тесты).
Визуальных изменений нет — чисто структурный проход. Проверено полным прогоном гейтов после каждого шага.
