1 GPS: модалка водителя — телефон и полный разбор транспорта FIX FEATURE

На странице GPS-отслеживания менеджер по водителям кликал по своему водителю — и открывалась почти пустая карточка. Запрос GET /v1/dispatchers/drivers/:id уходил и возвращал данные (в сети видно телефон, координаты, ТС), но панель их не показывала: телефон не выводился нигде, а транспорт схлопывался в одну строку. У водителя без маршрута/ТС середина модалки оставалась пустой.

Что сделано

  • Карточка телефона (иконка + ссылка tel:) — показывается, когда телефон доступен (для менеджера по водителям), скрыта для ограниченных ролей.
  • Карточка «Транспорт» с раздельными блоками Тягач и Прицеп: тип · модель · гос. номер · техпаспорт · владелец. Каждая строка рендерится только при наличии значения («если null — не показываем»).
  • Слияние ответа профиля с ростером теперь накладывает только непустые поля — разрежённый /drivers/:id больше не затирает данные о ТС/рейтинге, уже известные из ростера.
  • 10 новых ключей gpsTrackingPage.* добавлены во все 15 локалей.
Телефон +998941780618, статус в сети, и полный разбор ТС: тягач (MAN TGX, гос. номер, техпаспорт, владелец) + прицеп (Schmitz).

Почему середина была пустой

Панель рисует карточки условно: статус-сообщение, реалтайм-присутствие, маршрут, ТС, диспетчер. У водителя без маршрута и без единой строки о ТС оставалась только карточка «в сети», а телефон не выводился вовсе — хотя detail.phone уже был заполнен. Теперь телефон и структурированный разбор ТС закрывают эту пустоту.

Файлы: DriverDetailSidebar.tsx (карточка телефона + SpecRow + карточка «Транспорт»), useGpsDetailSelection.ts (наложение непустых полей), types.ts (truck/trailer получили model/owner).

2 Админ-панель «Рейсы»: водитель, тягач, «был онлайн», диспетчер FEATURE

В админской таблице рейсов (/admin/trips) по каждому рейсу были только базовые поля. Требовалось показать имя и телефон водителя, номер тягача, время последнего онлайна и — если у водителя есть диспетчер — его имя и телефон.

Сверено с живой спецификацией (base.yaml#/Trip): рейс несёт только плоские id (driver_id, driver_manager_id, …) плюс vehicle_snapshot — встроенных объектов водителя/диспетчера нет. Поэтому решено без изменений бэкенда, по устоявшемуся админскому паттерну: подгрузить списки /v1/admin/drivers и /v1/admin/dispatchers один раз и соединить по id (а не N+1 запрос на строку).
  • Водитель — имя + телефон (join по driver_id).
  • Номер тягача — из vehicle_snapshot.power_plate_number рейса (с запасным вариантом — гос. номер из карточки водителя).
  • Был онлайн — last_online_at водителя.
  • Диспетчер — имя + телефон через driver_manager_id (запасной путь — freelancer_id водителя); при отсутствии показывает «Нет данных».
  • Разбор снапшота/участников считается один раз в мемоизированном индексе — сортировка не перевыполняет base64-декод.
Новые колонки заполнены: имя+телефон водителя, номер тягача, «был онлайн» и диспетчер (у 3-й строки диспетчера нет — честное «Нет данных»). Новые чистые модули adminTripParticipants.ts + 19 юнит-тестов.

3 Создание груза скрыто для компании-перевозчика (CARRIER) FIX

Кабинет компании показывал вкладку «Грузы» и кнопку «Добавить груз» всем компаниям. Перевозчик (CARRIER) предлагает транспорт, а не груз, и постить груз не должен (бэкенд и так отклоняет создание с company_type_forbidden). Кнопка была завязана только на право cargo.create, которое у владельца «срабатывает по умолчанию» — так перевозчик видел кнопку, которая всё равно 403.

Что сделано

  • Вся вкладка «Грузы» скрыта для CARRIER через forbiddenForTypes: ['CARRIER'] в реестре секций — она исчезает из навигации, а deep-link ?tab=cargo откатывается на обзор. У SHIPPER/BROKER вкладка остаётся.
  • Дополнительно (защита в глубину) — чистый предикат companyTypeAllowsCargoCreation гейтит саму кнопку создания в секции.
  • Тип компании берётся как company_type_canonical ?? type — как и в остальном кабинете.
CARRIER (перевозчик): в группе «Операции» — Рейсы · Доска · Предложения, без «Грузы». Группа «Автопарк» присутствует.
SHIPPER (грузовладелец): «Грузы» на месте; «Автопарк» отсутствует (это уже существующее правило для грузовладельца).
Файлы: companyWorkspaceSections.ts (реестр), companyCargoForm.ts (companyTypeAllowsCargoCreation), CargoSection.tsx (гейт кнопки) + 2 теста реестра + 3 теста предиката.

4 «Поделиться» в «Моих грузах» диспетчера FIX

В таблице «Мои грузы» у менеджера по грузам не было способа поделиться собственным объявлением — кнопка «Поделиться» жила только в «Общих грузах», из-за чего приходилось идти в отдельный список.

  • Тот же компонент CargoShareMenu (то же модальное окно «Чат Sarbon», те же ссылки и эндпоинты) добавлен в колонку действий таблицы «Мои грузы» — кнопка на каждой строке.
  • Обёртка гасит всплытие клика, чтобы «Поделиться» не открывало карточку груза.
  • Скоуп — таблица (по тикету); карточный вид — единая обёртка-кнопка, куда вложенная кнопка была бы недопустимой (button-in-button), оставлено на будущее.
Табличный вид «Моих грузов» (роль «Менеджер по грузам») — иконка «Поделиться» в колонке действий каждой строки.
Открытое меню: Чат Sarbon · Копировать ссылку · WhatsApp · Telegram · Facebook · SMS · Почта · Instagram — идентично «Общим грузам».
